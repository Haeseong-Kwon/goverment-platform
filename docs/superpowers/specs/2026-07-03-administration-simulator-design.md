# 행정 시뮬레이터 이식 설계

## 목표

`행정 시뮬레이터 - 제품 명세서.md`를 단일 진실 원천으로 삼아 7개 화면, 공통 디자인 토큰, 도메인 데이터, B/C/F 상태 머신과 이중 승인 게이트를 기존 Next.js 앱에 이식한다.

## 아키텍처

- Next.js App Router의 `(simulator)` 라우트 그룹에 7개 URL을 구성한다.
- 기존 루트 레이아웃은 유지하되, simulator 레이아웃이 활성화된 경로에서는 기존 AOP Navbar/Footer를 렌더하지 않도록 경로 인식 셸을 둔다.
- 명세 전용 CSS 변수는 `.simulator-shell` 범위에 등록하여 기존 테마와 충돌하지 않게 한다.
- 공통 컴포넌트와 typed mock data를 `src/features/administration-simulator` 아래에 함께 배치한다.
- 정적 화면은 Server Component, B/C/F 및 승인·반려 입력이 필요한 화면은 최소 범위의 Client Component로 구현한다.

## 컴포넌트 경계

- `components/ui`: Card, Button, StatusPill, RoleBadge, StatCard, DonutChart, DocRow, ProgressBar, Stepper.
- `data`: 과제, 비목, 팀, 한도, 큐, 증빙, 사유코드.
- `logic`: 증빙 첨부 수와 한도 초과 기반 제출 검증, 큐 상세 매핑, 게이트별 허용 사유코드.
- `pages`: 명세 §3~§9의 각 화면.

## 상태와 데이터 흐름

- 온보딩은 `step: 1 | 2 | 3` 로컬 상태로 직접 이동과 이전/다음을 처리한다.
- 정산 제출은 `account`에 따라 증빙 목록을 파생하고, 누락 증빙 수 및 한도 초과 여부로 메시지와 제출 가능 여부를 계산한다.
- 검토 큐는 선택된 `q`로 상세 패널을 파생한다. 반려는 사유코드와 코멘트를 요구한다.
- 회계사 화면은 배정 목록과 게이트1 승인/반려만 노출한다. 대시보드와 통계 진입점은 만들지 않는다.

## 반응형

- 1440px에서는 명세 픽셀값을 그대로 사용한다.
- 컨테이너는 화면별 760/960/1100/1200/1320px max-width를 유지한다.
- 1024px 이하에서 2열/3열/4열을 단계적으로 축소하고, 768px 이하에서는 단일 열로 전환한다.
- 스테퍼와 데이터 표는 의미를 보존하면서 좁은 화면에서 세로 또는 카드형으로 재배치한다.

## 검증

- 순수 상태 계산 로직은 Vitest로 RED-GREEN 검증한다.
- ESLint와 production build를 실행한다.
- 브라우저에서 1440px, 768px, 390px 뷰포트로 각 경로를 열어 overflow, 문구, 상태 전환, 툴팁, disabled 상태를 확인한다.
- 마지막에 명세 §10.6 체크리스트를 항목별로 대조하고 미충족 항목을 보고한다.
