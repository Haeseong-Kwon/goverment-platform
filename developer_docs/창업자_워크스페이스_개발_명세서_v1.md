## 워크플로우 단계

**STEP 1. 진입 분기 (창업자 / 매니저)**

**✎ 디테일 (작성):** 첫 화면에서 역할별 경험을 완전 분리. 창업자는 무료 즉시 시작, 매니저는 기관 인증 절차로 진입.

**⚙ 기술 보강:** \[권장\] 단일 Next.js 앱 + role 기반 라우팅(/f/\*, /m/\*). 매니저 가입은 기관 코드 or 도메인 화이트리스트 승인제. \[사실\] 앱 2개 분리는 기각(컴포넌트 중복·배포 부담).

**STEP 2. 프로필 입력**

**✎ 디테일 (작성):** 포지션과 "함께할 팀원을 찾고 있어요" 의향 필드를 받되, 팀 탐색 UI는 제공하지 않음(공급 풀 사전 축적 목적).

**⚙ 기술 보강:** \[사실\] 팀빌딩 탐색은 스코프 아웃(W7 프리뷰만). 의향·찾는 포지션은 team_building_intent로 저장해 W7 대기 수요 집계에 사용.

**STEP 3. 지원사업 선택**

**✎ 디테일 (작성):** 예창패·초창패·모두의창업 등 멀티 선택. 이 선택이 D-day·자동 마일스톤·자격진단·추천의 데이터 원천.

**⚙ 기술 보강:** \[권장\] Program 마스터 테이블(사업명·연도·공고일·마감일·규정 룰셋 버전) + 운영자 어드민에서 일정 관리. \[가설\] K-Startup 공고 일정 자동 수집은 약관·안정성 확인 필요 - MVP는 운영자 수기 입력.

**STEP 4. 아이템 입력 + 팀원 초대**

**✎ 디테일 (작성):** 아이템 한 줄 + 업종. 초대 링크로 팀원(pre_founder)이 같은 PrepTeam에 합류.

**⚙ 기술 보강:** \[권장\] 초대 링크는 서명 토큰(만료 7일·팀 귀속). 초대 수락 이벤트는 바이럴 계수 KPI로 로깅.

**STEP 5. 온보딩 완료 → 자격진단 자동 실행**

**✎ 디테일 (작성):** 첫 세션에서 "아하 모먼트"를 전달하기 위해 완료 즉시 W1을 자동 실행한다.

**⚙ 기술 보강:** \[권장\] 온보딩 답변을 W1 입력 폼에 프리필. 진단 소요 3초 내 목표(룰 기반이라 LLM 불필요 - W1 참조).

**▶ 다음 단계로 넘기는 산출물:** {user, prep_team, programs\[\], item, team_building_intent} → W1 자동 실행, W2 마일스톤 생성

## 이 모듈 전반 메모

**⚙ 기술 보강:** 예외: 지원사업 미선택 가입 허용 - 이 경우 W2 캘린더·TODO는 empty state("사업을 선택하면 일정을 자동 생성") 노출. 소셜 로그인 실패 시 이메일 폴백.

# W1. 자격 사전진단 + 지원사업 추천 (P4-1)

_프로필·이력 입력 → 룰 엔진 평가 → 자격 적합도 리포트 + 지원 가능 사업 추천._

| **입력 (Input)**                                                                             | **최종 산출물 (Output)**                                                            |
| -------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| 창업 이력(사업자등록·폐업), 동일사업 기수혜 여부, 대표자 요건(연령·재직 상태), 공동대표 여부 | 자격 적합도 리포트(점수·결격 위험·미확인 항목·다음 할 일), 지원 가능 사업 추천 카드 |

## 워크플로우 단계

**STEP 1. 자격 문항 입력**

**✎ 디테일 (작성):** 서류 실격은 노력이 아니라 정보 문제 - 신청 전에 결격 사유를 적발하는 것이 이 모듈의 존재 이유.

**⚙ 기술 보강:** \[권장\] 문항은 룰셋에서 동적 생성(사업마다 요건 상이). 민감 정보(주민번호 등)는 수집하지 않음 - 여부(boolean)만 받는다.

**STEP 2. 룰 엔진 평가**

**✎ 디테일 (작성):** 공고문 조항 기반의 확정적 판정이 필요. AI의 창의적 해석이 개입되면 안 되는 영역.

**⚙ 기술 보강:** \[사실\] LLM 미사용 - 결정적 룰 엔진으로 구현(오탐 시 신뢰 붕괴 방어). \[권장\] 룰셋은 코드와 분리된 JSON(사업×연도 버전 관리, 예: yechang_2026.json). 규정 개정 시 배포 없이 룰셋만 교체.

**STEP 3. 리포트 생성 (A1 리포트 문법)**

**✎ 디테일 (작성):** 점수 → 기여 분해 → 판단 근거 → 미확인 데이터 → 다음 할 일. "아직 확인하지 못한 데이터" 섹션은 절대 생략 금지.

**⚙ 기술 보강:** \[사실\] 공통 컴포넌트 ScoreRing / ContributionBars / ReasonList / NextActions 사용(디자인 확정본). 근거에는 룰셋 조항 ID를 각주로 남겨 재현 가능하게.

**STEP 4. 지원 가능 사업 교차 추천**

**✎ 디테일 (작성):** 같은 입력으로 타 사업 룰셋을 교차 평가해 "이 조건이면 이 사업도 가능"을 제시 - 단순 키워드 매칭 서비스(웰로 등)와의 차별점.

**⚙ 기술 보강:** \[권장\] 전 룰셋 일괄 평가 → 적합/확인필요/부적합 3분류, RecommendationCards로 출력. \[캘린더에 추가\] 클릭 시 W2 Program 구독 + 마일스톤 생성.

**▶ 다음 단계로 넘기는 산출물:** {eligibility_score, blockers\[\], unchecked\[\], recommended_programs\[\]} → W2(캘린더 추가), 홈 위젯

## 이 모듈 전반 메모

**✎ 디테일 (작성):** 사람 개입/검수 지점 (작성): 룰셋 작성·갱신은 운영자(기획) 책임. 공고 개정 모니터링 프로세스 필요.

**⚙ 기술 보강:** 예외: 룰셋 미보유 사업 선택 시 "진단 준비 중" 표시 + 대기 신청 수집(룰셋 우선순위 결정 데이터). 판정 불가 항목은 단정하지 않고 "확인 필요" 잔류.

**⚙ 기술 보강:** \[사실\] 모든 출력에 "AI/자동 판정은 참고용, 최종 자격은 공고문 기준" 고지 고정. \[권장\] 이벤트: diagnosis_complete, recommendation_add_to_calendar.

# W2. 워크스페이스 코어 - 홈·팀 TODO·마감 캘린더 (P2·P3·P7)

_지원사업 마감일 역산 → 자동 마일스톤 생성 → 팀 칸반 + 캘린더 + 홈 집계._

| **입력 (Input)**                                         | **최종 산출물 (Output)**                                  |

| PrepProject(사업·마감일), 커스텀 TODO, 팀원 배정, 코멘트 | 자동 마일스톤 칸반, D-day 캘린더 + 알림, 홈 대시보드 위젯 |

## 워크플로우 단계

**STEP 1. 자동 마일스톤 생성 엔진**

**✎ 디테일 (작성):** "뭘 언제까지 해야 하는지"를 도메인 지식으로 내장 - 사업계획서 초안 D-14, 증빙 준비 D-10, 발표 리허설 D-7, 최종 제출 D-1. 일반 TODO 앱과의 결정적 차별점.

**⚙ 기술 보강:** \[권장\] milestone_template 테이블(program_type별 오프셋). 마감일 변경 시 재계산. 자동 생성 카드는 삭제 대신 "숨기기"(감사 추적 유지). auto 뱃지로 커스텀과 구분.

**STEP 2. 팀 칸반 + 코멘트 스레드**

**✎ 디테일 (작성):** 커뮤니케이션은 업무 객체(TODO·문서)에 종속시킨다. 실시간 팀 채팅은 만들지 않는다 - 카톡과 경쟁 불가, 개발비 최대.

**⚙ 기술 보강:** \[사실\] 채팅 스코프 아웃 확정. \[권장\] 코멘트는 폴링/refetch로 충분(웹소켓 불필요), 멘션 시에만 알림. 칸반 3열 + 드래그 정렬.

**STEP 3. 마감 캘린더 + 알림**

**✎ 디테일 (작성):** 복수 사업의 마감 충돌을 한눈에. 알림이 재방문 리듬을 만든다(리텐션 채널).

**⚙ 기술 보강:** \[가설\] 알림 채널 비용 비교 필요 - 이메일(무료급) vs 카카오 알림톡(건당 과금). MVP는 이메일 + 브라우저 푸시, 알림톡은 2차. D-7/D-3/D-1 자동 발송.

**STEP 4. 홈 대시보드 집계**

**✎ 디테일 (작성):** 로그인 10초 안에 "오늘 할 일"이 보여야 한다.

**⚙ 기술 보강:** \[권장\] 단일 집계 API(GET /f/home): 최근접 D-day, 미완료 TODO 상위 3, 진단 요약, 팀 진행률. N+1 방지 위해 서버 집계.

**▶ 다음 단계로 넘기는 산출물:** 주간 활동 데이터(WAU·TODO 완료율) → KPI 리텐션 축. 마일스톤 상태 → 홈·캘린더 동기화

## 이 모듈 전반 메모

**✎ 디테일 (작성):** 사람 개입/검수 지점 (작성): 마일스톤 템플릿 정의는 기획 책임(사업별 실제 준비 순서 반영).

**⚙ 기술 보강:** 예외: 마감일 경과 사업은 자동 아카이브 + "다음 공고 알림 받기" 전환 제안. D-day 컬러 로직 고정: D-8+ slate / D-7~4 amber / D-3- red.

**⚙ 기술 보강:** \[사실\] 진단 점수·준비 이력은 이관 후에도 매니저에게 비노출(가시성 절대 규칙). 매니저는 검증 통과 제출 건만 조회 - DB 레벨(RLS/미들웨어) 강제, 프론트 숨김만으로 처리 금지.

**▶ 다음 단계로 넘기는 산출물:** founder 워크스페이스 → 기존 정산 모듈(\[B\]\[C\]\[D\]) → 매니저 대시보드(\[E\]\[F\]) 파이프라인 연결
